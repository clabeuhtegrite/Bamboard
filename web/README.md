# Project website + web installer

The Bamboard project site, published to GitHub Pages at
**https://clabeuhtegrite.github.io/Bamboard/**. It does two jobs:

1. **Showcase** — a single-page presentation of the project: hero, features, an
   interactive walk-through of the six device screens (using the demo
   screenshots), the Bambuddy architecture, the hardware/BOM, and install &
   update info.
2. **Browser flasher** — built on
   [ESP Web Tools](https://esphome.github.io/esp-web-tools/). A visitor on a
   Chromium-based desktop browser (Chrome / Edge / Opera) clicks **Connect**,
   picks the board's USB serial port, and the page flashes the full firmware
   image over Web Serial. No CLI, no toolchain.

The page is static and works without JavaScript; `app.js` is progressive
enhancement (version badge, the tabbed screen showcase, lightbox, copy button,
mobile nav, scroll reveals).

## Languages

The page ships in English and translates itself **client-side** into the
languages the firmware supports — English, Spanish, French, Portuguese and
German (mirroring `firmware/src/ui/i18n.cpp`) — with **English as the
fallback**. No third-party widget: the dictionaries live in `i18n.js`, and the
English DOM is both the source and the fallback, matching the project's
local-first, no-external-cloud stance (a no-JS visitor still gets English).

The language is picked from `?lang=xx` (then remembered in `localStorage`) →
the browser's `Accept-Languages` → English, and a switcher in the nav lets
visitors override it. Translatable nodes are tagged in `index.html` with
`data-i18n` (textContent), `data-i18n-html` (innerHTML, for phrases with inline
markup) and `data-i18n-attr` (attributes like `alt` / `aria-label`); brand and
acronym terms (Bamboard, Bambuddy, AMS, HMS…), units and the device-screen tab
names — the screenshots are rendered in English — stay verbatim. The JSON
dictionary block in `i18n.js` is delimited by `i18n:json` markers so CI can
validate it.

## Files

Committed:

- `index.html` — the page (showcase + the `<esp-web-install-button>`).
- `styles.css` — Bamboard-themed styles (palette mirrors `firmware/src/config.h`).
- `app.js` — progressive-enhancement script.
- `i18n.js` — self-hosted client-side translation (auto-detects the visitor's
  language, English fallback). See **Languages** below.
- `assets/favicon.svg` — site icon (the Live-screen progress-ring mark).
- `assets/og-image.svg` — social-share card source; **rasterised to
  `og-image.png` by CI** at deploy time (crawlers want PNG/JPG, not SVG).
- `manifest.json` — ESP Web Tools manifest. The committed copy is a dev
  placeholder; the **published** one is regenerated by CI with the release
  version. It points at `bamboard-factory.bin` at offset `0x0`.

Pulled in at deploy time (**not committed**):

- `bamboard-factory.bin` — the merged factory image, downloaded from the latest
  release and served **same-origin** (GitHub release assets send no CORS
  headers, so the browser can't fetch the bin cross-origin).
- `screenshots/*.png` — the six demo screens, rendered by the host sim
  (`SIM_FIXTURES_ONLY=1`) from a deterministic dataset, so they always match the
  current UI. See `docs/screenshots/README.md`.
- `screenshots/hero.png` — the hero: `scripts/compose_hero.sh` warps the rendered
  `live.png` onto TomE1337's case photo (`docs/screenshots/case.jpg`, CC BY-NC-SA
  4.0) in perspective. A derivative of that photo, so likewise CC BY-NC-SA.
- `assets/og-image.png` — rasterised from `assets/og-image.svg`.

## How it ships

Two workflows, decoupled:

- **`release.yml`** (on every `v*` tag) builds the firmware, runs
  `esptool merge_bin` → `bamboard-factory.bin`, and publishes the GitHub Release
  (app `firmware.bin` for OTA + the factory bin + `manifest.json`).
- **`pages.yml`** assembles this `web/` folder into the Pages site: it copies the
  static files, pulls the hero render from `docs/`, rasterises the OG image,
  downloads the latest release's factory bin + version (regenerating
  `manifest.json`), and renders the demo screenshots with the sim. It runs on a
  release (`release: published`), on pushes that touch `web/` or `pages.yml`, and
  on manual dispatch — so editing the page redeploys it without a firmware
  rebuild, and cutting a release refreshes the firmware it offers.

So the web installer always offers the firmware from the most recent release,
and the showcase always reflects the current UI.

## Browser support

Web Serial is **Chromium desktop only** (Chrome / Edge / Opera). Firefox,
Safari and mobile browsers can't flash; the page tells them to use a supported
browser or flash `bamboard-factory.bin` by hand with esptool.
